04
Los 4 problemas

Lo que la IA hace mal

01
Falta de contexto

Al empezar cada sesión, la IA no sabe nada de tu base de código. Cero.

02
Alucinaciones

Inventa APIs, métodos y dependencias que no existen. Y te lo dice convencida.

03
Indeterminismo

El mismo prompt puede dar resultados diferentes cada vez que lo ejecutas.

04
Difícil de verificar

Generamos código más rápido de lo que nuestra carga cognitiva permite revisar.

05
El flujo que ya conocemos

Separar fases no es opcional con IA

01
Investigar
Contexto, restricciones, decisiones
→
02
Planificar
Diseño, pasos pequeños, criterios
→
03
Implementar
Sesión limpia, contexto preciso
→
04
Validar
Tests, revisión, feedback real

Si colapses investigación, planificación e implementación en una sola sesión, sacrificas coherencia por velocidad.
La ventana de contexto no es infinita — gestiona el contexto de forma agresiva.

06
Ya sabemos lo que funciona

El cuello de botella nunca fue
el número de líneas por minuto

01
Story Splitting

Partir trabajo en piezas verticales y desplegables. Con IA esto es aún más crítico.

02
Hamburger Method

Entrega de valor de extremo a extremo antes de completar la funcionalidad completa.

03
Small Safe Steps

Cada paso debe ser reversible. Que el agente genere mucho no significa que despliegues todo junto.

04
Testing avanzado

Mutation testing, Acceptance Testing, Architectural Testing — amplían el ciclo de feedback.

El ciclo de feedback es acortar el tiempo entre "escribir código" y "saber si es correcto".

07
Diseña para supervisión

No puedes revisar cada línea.
Diseña sistemas que capturen errores.

Guardarraíles técnicos

  • Hooks: linter automático en cada tool use
  • Tests de arquitectura: verifican estructura, no solo funcionalidad
  • Acceptance Testing: el comportamiento esperado definido antes
  • CI/CD maduro: si no es fiable con humanos, será un desastre con agentes
  • Permisos del agente: define qué puede hacer sin aprobación y qué no

Checklist antes de mergear

  • ¿Esto respeta mi dominio o solo suena bien?
  • ¿Los tests prueban algo real?
  • ¿Hay observabilidad de ese caso de uso?
  • ¿Podría borrar este código sin romper nada?

La transformación

¿Qué cambia?

Tu rol

  • ✕ Ya no te pagan por escribir código
  • ✕ Tampoco por usar IA
  • ✓ Te pagan por decidir qué NO debe entrar en producción
  • ✓ Te pagan por proteger el sistema

El entorno

  • → La herramienta de escritura de código
  • → La velocidad de generación
  • → La barrera económica del testing riguroso
  • → El formato de la documentación (CLAUDE.md, skills)
  • → Quién escribe el código (humano vs. agente)
10
Lo que importa

¿Qué permanece — y cobra más fuerza?

01
Arquitectura y diseño simple

La IA genera piezas. El reto es integrar y mantener a lo largo del tiempo.

02
Pasos pequeños y seguros

El cuello de botella nunca fue generar código — fue acortar el ciclo de feedback.

03
Definir el comportamiento antes de implementar

Si no lo defines tú, la IA hará unos tests perfectos para el código que ella misma decidió.

04
Autonomía ajustada al riesgo

No todo necesita el mismo nivel de rigor. Pero cada nivel necesita sus guardarraíles.

05
Trabajo en equipo y comunicación clara

La IA amplifica lo que hay. Si estamos desalineados, amplificará el caos.

06
We ARE uncovering

Esto no es una revolución que lo reemplaza todo. Es la siguiente iteración de un proceso que no termina.
